    def add_baseline_variation(coords, amplitude=2.0, frequency=0.02):
        """Add natural wavy baseline like real paper writing"""
        coords = np.copy(coords)
        x = coords[:, 0]
        # Sine wave + low-frequency noise
        wobble = amplitude * np.sin(frequency * x) + \
                 np.random.normal(0, amplitude * 0.3, len(x))
        coords[:, 1] += wobble
        return coords

    4. Robotic connections — Ligatures between letters


Fixes for Maximum Realism:

TECHNICAL DETAILS: CUSTOM LSTM IMPLEMENTATION
──────────────────────────────────────────────

    Function: lstm_cell_call(inputs, state, num_units, scope)
    
    Purpose: Replace tf1.nn.rnn_cell.LSTMCell which is unavailable in Keras 3
    
    Implementation:
    
        def lstm_cell_call(inputs, state, num_units, scope):
            c, h = state.c, state.h
            input_size = inputs.get_shape().as_list()[-1]
            
            with tf1.variable_scope(scope, reuse=tf1.AUTO_REUSE):
                # Combined gates: input, forget, cell, output
                kernel = tf1.get_variable(
                    'kernel',
                    shape=[input_size + num_units, 4 * num_units],
                    initializer=tf1.initializers.glorot_uniform()
                )
                bias = tf1.get_variable(
                    'bias',
                    shape=[4 * num_units],
                    initializer=tf1.zeros_initializer()
                )
                
                gate_inputs = tf.matmul(tf.concat([inputs, h], 1), kernel)
                gate_inputs = tf.nn.bias_add(gate_inputs, bias)
                
                # Split into 4 gates
                i, j, f, o = tf.split(gate_inputs, 4, axis=1)
                
                # Apply activations
                forget_bias = 1.0
                new_c = (c * tf.sigmoid(f + forget_bias) + tf.sigmoid(i) * tf.tanh(j))
                new_h = tf.tanh(new_c) * tf.sigmoid(o)
                
                new_state = LSTMStateTuple(c=new_c, h=new_h)
                return new_h, new_state
    
    Variable Naming:
    • Uses 'kernel' and 'bias' to match original tf.nn.rnn_cell.LSTMCell naming
    • Scope names: 'lstm_cell', 'lstm_cell_1', 'lstm_cell_2'
    • Matches pretrained checkpoint: checkpoints/model-17900

Overview
────────
Great ask — these 8 points are basically the difference between a demo that sometimes works and a product that runs reliably.

1. Fix API contract mismatches (critical)
   ──────────────────────────────────────
   Right now one layer sends `style_id`, but another layer expects `style`. Because of that mismatch, selected styles can be ignored and defaults get used.

   Why this matters:
   users think style selection works, but backend may always render style `0`.

   Improvement:
   standardize one request/response contract across client → Node → Python (same field names, same types), or add explicit mapping at one gateway.

2. Unify style endpoints and data shape
   ─────────────────────────────────────
   Your Node side expects one JSON format (`{ styles: [...] }`, fields like `id/chars`), while Python returns a different one (`style_id`, `metadata`, sometimes raw list).

   Why this matters:
   style list/load/delete flows can break silently or show empty data.

   Improvement:
   define one canonical style schema and convert all endpoints to it (including `GET /styles`, `GET /styles/:id`, `DELETE /styles/:id`).

3. Replace placeholder style extraction with real inference
   ─────────────────────────────────────────────────────────
   In `hwt_wrapper.py`, style extraction is currently random (`np.random.randn(...)`), not truly extracted from the uploaded handwriting image.

   Why this matters:
   custom style is fake right now; same input can give inconsistent output.

   Improvement:
   run actual HWT inference pipeline for image preprocessing + model forward pass + deterministic embedding save.

4. Complete pending core features (Step 6–9)
   ───────────────────────────────────────────
   Mongo persistence, upload-driven style extraction, export, and custom style management are still incomplete. Export buttons are currently placeholders.

   Why this matters:
   users can generate preview, but real product workflows (save, reuse, export, account-based data) are incomplete.

   Improvement:
   wire full end-to-end flows: save styles/docs in DB, connect upload to style creation, and implement real PNG/PDF/SVG export path.

5. Remove hardcoded localhost URLs
   ────────────────────────────────
   Some frontend auth calls are hardcoded to `http://localhost:5000`.

   Why this matters:
   works locally, breaks in staging/production and different environments.

   Improvement:
   move all URLs to environment variables (`VITE_API_URL`, etc.) and use one shared API config.

6. Stabilize Python dependencies/environment
   ──────────────────────────────────────────
   Python dependency files are inconsistent (one minimal, one old TF1 pin), and runtime compatibility is fragile.

   Why this matters:
   works on one machine problems, startup failures, import/version conflicts.

   Improvement:
   create one reproducible dependency lock (single supported Python version + pinned compatible package set) and document exact setup.

7. Add resilience between Node and ML service
   ───────────────────────────────────────────
   Current ML calls lack timeout/retry/circuit-breaker/concurrency controls.

   Why this matters:
   if Python slows or hangs, requests can pile up and freeze user-facing APIs.

   Improvement:
   add request timeouts, bounded retries for safe endpoints, failure-fast behavior, and concurrency limits on `/generate`.

8. Improve dev/prod orchestration
   ───────────────────────────────
   Running requires 3 manual terminals, and `vercel.json` only covers frontend deployment.

   Why this matters:
   setup is error-prone, and production architecture is unclear for backend/ML service.

   Improvement:
   add single-command local startup with health checks, and document/deploy frontend, Node API, and ML service as explicit separate services.

Practical implementation order
──────────────────────────────
`1 → 2 → 5 → 6 → 7 → 4 → 8 → 3` for the fastest stability gains first.
